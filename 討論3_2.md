## 三、系統目標

| 項目 | 說明 |
|------|------|
| A | 使用者於 MyAppDesign 前端輸入 `10.10.10.1` 的帳號密碼，送至後端 API，由 EasyAPI 驗核身份。 |
| B | 驗證成功後，後端將該帳密短暫交給 Session Manager，由 Playwright 建立專用 BrowserContext，登入 `http://10.10.10.1/login.php`，成功後進入 `menu.php`。 |
| C | 站台登入成功後進入 `menu.php`，該 Playwright session 進入可操作狀態，等待使用者後續指令。 |
| D | 使用者指示最終希望可透過智能體（如 OpenCode / Hermes / OpenAI API）進行語意解析，轉成結構化 command，再交給固定 action 執行。此項屬後期規劃，第一版不實作完整 AI Resolver。 |
| E | 智能體與 Session Manager 部署於主機 `10.10.10.1`，但第一版僅實作 Session Manager；智能體層待後期導入。 |

## 四、整體架構總覽

建議架構如下。注意：`AI Resolver` 為後期導入元件，不屬於第一版 MVP。

```text
[MyAppDesign 前端]
  使用者輸入 10.10.10.1 帳密
        ↓
[MyAppDesign 後端 API]
  接收帳密，呼叫 EasyAPI 驗核
        ↓
[EasyAPI 10.10.10.1:3001]
  驗核帳密是否有效
        ↓
[Session Manager 10.10.10.1:3002]
  建立 Playwright BrowserContext
  使用同一組帳密登入 login.php
  成功後進入 menu.php
        ↓
[Playwright Session]
  保持該使用者專屬瀏覽器狀態
        ↓
[Playwright 探索操作]
  初期先確認可操作性
        ↓
[browser.actions.ts]
  將穩定流程逐步封裝成固定函式
        ↓
[action.registry.ts]
  只允許呼叫已註冊 action
        ↓
[AI Resolver，後期導入]
  將自然語言轉成 intent + params
```

核心原則：

```text
第一版不導入完整 AI Resolver
AI 後期只負責理解語意
系統負責驗證與派工
Playwright 只執行可控操作
browser.actions.ts 逐步累積已驗證成功的功能
Session Manager 負責隔離、狀態、逾時與錯誤恢復
```

## 五、登入流程修正版

使用者登入 MyAppDesign 時，輸入的不是另一組 MyAppDesign 專用帳密，而是 `10.10.10.1` 站台的帳號密碼。

修正後登入流程：

```text
使用者在 MyAppDesign 前端輸入 10.10.10.1 的帳號密碼
  ↓
MyAppDesign 後端 API 接收 username / password
  ↓
後端呼叫 EasyAPI 驗核身份
  ↓
驗核成功後，後端將同一組帳密短暫交給 Session Manager
  ↓
Session Manager 使用 Playwright 開啟 http://10.10.10.1/login.php
  ↓
Playwright 填入帳密並送出登入表單
  ↓
登入成功後進入 menu.php
  ↓
建立該使用者專屬 Playwright Session
  ↓
MyAppDesign 回傳 appSessionId 給前端
```

前端後續所有操作都使用 `appSessionId`，不再重複傳送密碼。

## 六、帳密處理原則

### 6-1 前端原則

前端可以讓使用者輸入 `10.10.10.1` 的帳密，但不可保存帳密。

不可執行：

```text
不可存 localStorage
不可存 sessionStorage
不可寫入 cookie
不可在 console.log 印出
不可在錯誤訊息中回傳
不可在 URL query string 中傳遞
```

前端登入成功後，只保存 MyAppDesign 自己發出的：

```text
appSessionId
或應用層 session token
```

### 6-2 後端原則

後端可以在登入流程中短暫使用帳密。

```text
1. 接收 username / password
2. 呼叫 EasyAPI 驗核
3. 驗核成功後交給 Session Manager 建立 Playwright session
4. Playwright 登入完成後即不再保存 password
```

Session Manager 內部不應長期保存明文密碼。

建議 session 只保留：

```ts
type ManagedSession = {
  sessionId: string
  userId: string
  context: BrowserContext
  page: Page
  state: SessionState
  createdAt: Date
  lastActiveAt: Date
  currentUrl?: string
}
```

不建議長期保留：

```ts
{
  username: string,
  password: string
}
```

若未來需要自動重新登入，可再評估加密保存或重新要求使用者登入。第一階段建議採「session 失效後要求使用者重新登入」。

## 七、各層技術選型

### 7-1 身份驗核層：EasyAPI

使用者登入請求先送至 EasyAPI 進行身份驗核。

角色定位：

```text
EasyAPI 負責判斷這組 10.10.10.1 帳密是否有效
```

建議：

```text
驗核成功後回傳 userId / name / role 等必要資訊
不要回傳多餘敏感資料
不要把密碼寫入 log
驗核失敗時回傳明確錯誤碼，例如 INVALID_CREDENTIALS
```

### 7-2 瀏覽器自動化層：Playwright

選用 Playwright 的理由：

| 能力 | 說明 |
|------|------|
| 多 BrowserContext | 每個 context 可視為獨立 session，互不干擾。 |
| headless / headed | 開發時可用 headed 除錯，上線後改 headless。 |
| 表單操作 | 可填寫 input、點擊 button、等待頁面跳轉。 |
| Trace / Screenshot | 後期可用於錯誤排查。 |

初期登入測試範例：

```ts
import { chromium } from 'playwright'

async function main() {
  const browser = await chromium.launch({ headless: false })
  const context = await browser.newContext()
  const page = await context.newPage()

  await page.goto('http://10.10.10.1/login.php')

  await page.fill('#username', process.env.TEST_USERNAME!)
  await page.fill('#password', process.env.TEST_PASSWORD!)
  await page.click('button[type="submit"]')

  await page.waitForURL('**/menu.php')

  console.log('login success:', page.url())

  await page.pause()
}

main()
```

> 注意：上方 selector 僅為示意，實際需依 `login.php` HTML 欄位名稱調整。

### 7-3 AI 語意解析層（後期導入，第一版不實作）

本節僅保留最終架構方向。舊版 `討論3.md` 中 AI 與 Session Manager 同期導入的即期規劃已廢除。

AI 層最終目標是：

```text
自然語言 → 結構化 command
```

例如：

```text
使用者：幫我查 2026 年 6 月報表
```

轉成：

```json
{
  "intent": "query_report",
  "params": {
    "year": 2026,
    "month": 6
  }
}
```

可選方案：

| 方案 | 說明 | 適用情境 |
|------|------|----------|
| Ollama + Hermes-3 | 本地 LLM，提供 REST API | 無法連外網、要求低延遲或資料不出內網 |
| OpenAI API | 雲端模型，語意理解能力強 | 可連外網、需要更強語意理解 |
| OpenCode | 支援工具調用與程式化協作 | 需要 function calling 或任務代理能力 |

重要原則：

```text
第一版不導入完整 AI Resolver
AI 不直接操作瀏覽器
AI 不產生 selector
AI 不產生 Playwright code
AI 不直接決定點擊哪個按鈕
AI 只產生 intent + params
```

### 7-4 多 Session 管理層

Session Manager 是本系統核心，部署於 `10.10.10.1:3002`。

職責：

```text
建立 BrowserContext
保存 userId 與 appSessionId 對應
維護 page 狀態
執行登入
接收 command
派發給 browser.actions.ts
管理 READY / BUSY / ERROR 等狀態
處理逾時與關閉 session
```

## 八、browser.login.ts 建議獨立

登入是所有自動化操作的基礎，建議獨立成 `browser.login.ts`。

```ts
import type { Page } from 'playwright'

export async function loginToTargetSite(
  page: Page,
  credentials: {
    username: string
    password: string
  }
) {
  await page.goto('http://10.10.10.1/login.php', {
    waitUntil: 'domcontentloaded'
  })

  await page.fill('#username', credentials.username)
  await page.fill('#password', credentials.password)

  await Promise.all([
    page.waitForNavigation({ waitUntil: 'domcontentloaded' }),
    page.click('button[type="submit"]')
  ])

  if (!page.url().includes('menu.php')) {
    throw new Error(`LOGIN_FAILED_CURRENT_URL=${page.url()}`)
  }

  const hasMenu = await page.locator('text=主選單').count()
  const hasLogout = await page.locator('text=登出').count()

  if (!hasMenu && !hasLogout) {
    throw new Error('LOGIN_FAILED_MENU_NOT_READY')
  }

  return {
    ok: true,
    url: page.url()
  }
}
```

實作時應依實際頁面調整：

```text
username 欄位 selector
password 欄位 selector
登入按鈕 selector
登入成功後 URL
登入成功後的關鍵文字
失敗訊息 selector
是否存在 iframe / frame
```

## 九、browser.actions.ts 的定位

`browser.actions.ts` 是最終目標，但不應在第一階段就要求完整設計所有功能。

較合理的定位是：

```text
browser.actions.ts = 已驗證成功的 Playwright 操作函式集合
```

而不是：

```text
browser.actions.ts = 預先設計完整的所有功能清單
```

開發方式應採：

```text
先用 Playwright 直接測試操作
  ↓
確認某個流程能穩定執行
  ↓
將該流程封裝成函式
  ↓
放入 browser.actions.ts
  ↓
逐步形成固定函式庫
```

## 十、探索函式到固定函式的演進

### 10-1 初期：探索函式

初期可以建立較彈性的探索函式，協助確認 menu.php 的操作流程。

```ts
import type { Page } from 'playwright'

export async function goMenu(page: Page) {
  await page.goto('http://10.10.10.1/menu.php', {
    waitUntil: 'domcontentloaded'
  })

  return {
    ok: true,
    action: 'goMenu',
    url: page.url()
  }
}

export async function clickMenuItem(page: Page, text: string) {
  await page.click(`text=${text}`)

  await page.waitForLoadState('domcontentloaded').catch(() => {})

  return {
    ok: true,
    action: 'clickMenuItem',
    text,
    url: page.url()
  }
}
```

探索函式適合用來確認：

```text
選單文字
頁面跳轉
frame / iframe
表單欄位
查詢按鈕
結果表格
錯誤訊息
```

### 10-2 穩定後：固定函式

當某個功能流程確認穩定後，再封裝成正式函式。

```ts
import type { Page } from 'playwright'

export async function queryReport(
  page: Page,
  params: {
    year: number
    month: number
  }
) {
  await page.click('text=報表查詢')
  await page.fill('#year', String(params.year))
  await page.fill('#month', String(params.month))
  await page.click('text=查詢')

  await page.waitForLoadState('networkidle').catch(() => {})

  return {
    ok: true,
    action: 'queryReport',
    params,
    url: page.url()
  }
}
```

此時 `queryReport()` 才是可供後續 AI Resolver 或 Command Registry 使用的正式 action。

## 十一、Action Registry 設計

當 `browser.actions.ts` 累積多個穩定函式後，再建立 `action.registry.ts`。

```ts
import { queryReport } from './browser.actions'

export const actionRegistry = {
  query_report: queryReport
}
```

執行時：

```ts
const action = actionRegistry[command.intent]

if (!action) {
  throw new Error(`UNKNOWN_ACTION: ${command.intent}`)
}

const result = await action(session.page, command.params)
```

Action Registry 的作用：

```text
限制 AI 或前端只能呼叫已允許的功能
避免任意操作瀏覽器
讓每個功能都有固定入口
方便測試與權限控管
方便未來加入角色權限
```

## 十二、Session Manager 初期設計

初期可先使用記憶體 Map 管理 session。

```ts
type SessionState =
  | 'CREATING'
  | 'LOGGING_IN'
  | 'READY'
  | 'BUSY'
  | 'EXPIRED'
  | 'ERROR'
  | 'CLOSED'

type ManagedSession = {
  sessionId: string
  userId: string
  context: BrowserContext
  page: Page
  state: SessionState
  createdAt: Date
  lastActiveAt: Date
  currentUrl?: string
}
```

操作流程：

```text
取得 session
  ↓
確認 session.state === READY
  ↓
將狀態改為 BUSY
  ↓
執行 Playwright 操作
  ↓
更新 lastActiveAt
  ↓
將狀態改回 READY
```

若操作失敗：

```text
記錄錯誤
截圖
將狀態改為 ERROR 或 EXPIRED
必要時關閉 context
回傳明確錯誤碼
```

Session 生命週期管理需考量：

```text
閒置逾時自動釋放，建議 30 分鐘
定期 heartbeat 或狀態檢查
發現被導回 login.php 時標記 EXPIRED
必要時關閉 context 並要求重新登入
```

## 十三、API 設計建議

初期 API 不需要太複雜，可先提供：

```http
POST /api/login
POST /api/command
GET /api/session/:sessionId/status
POST /api/session/:sessionId/close
```

### 13-1 登入 API

```http
POST /api/login
```

Request：

```json
{
  "username": "user001",
  "password": "******"
}
```

後端流程：

```text
1. 呼叫 EasyAPI 驗核帳密
2. 驗核失敗則回傳 401
3. 驗核成功則建立 Playwright BrowserContext
4. 使用同一組帳密登入 login.php
5. 成功進入 menu.php
6. 回傳 appSessionId
```

Response：

```json
{
  "ok": true,
  "appSessionId": "sess_xxx",
  "user": {
    "id": "user001",
    "name": "王小明"
  }
}
```

### 13-2 指令 API

第一階段建議先接受結構化指令，不直接接自然語言。

```http
POST /api/command
```

Request：

```json
{
  "appSessionId": "sess_xxx",
  "intent": "click_menu_item",
  "params": {
    "text": "報表查詢"
  }
}
```

處理流程：

```text
1. 驗證 appSessionId 是否有效
2. 取得對應 ManagedSession
3. 確認 session.state === READY
4. 檢查 intent 是否存在於 action.registry.ts
5. 執行對應 action
6. 回傳操作結果
```

待 Playwright 操作流程穩定後，再導入 AI Resolver 讓使用者輸入自然語言。

## 十五、AI Resolver 導入時機（後期階段）

AI Resolver 應延後導入。舊版 `討論3.md` 中 AI 即期參與操作流程的規劃，在本版中已明確廢除。

不建議一開始就接 AI，原因：

```text
AI 可能解析正確，但瀏覽器操作尚未穩定
瀏覽器操作錯誤時，難以判斷是 AI 問題還是 Playwright 問題
早期除錯成本會過高
```

建議導入順序：

```text
1. Playwright 登入成功
2. menu.php 基本操作成功
3. 第一個固定 action 成功
4. 多 session 管理穩定
5. 再加入 AI Resolver
```

AI Resolver 回傳格式建議：

```ts
type ResolvedCommand = {
  intent: string
  confidence: number
  params: Record<string, string | number | boolean>
  missingFields?: string[]
}
```

若信心不足或參數不足：

```ts
if (command.confidence < 0.75) {
  return {
    status: 'need_confirmation',
    message: '我不確定你要執行哪個功能，請再補充。'
  }
}
```

## 十六、補強建議

### 16-1 第一版後必補

| 補強項目 | 建議技術 | 理由 |
|----------|----------|------|
| 操作指令 Queue | BullMQ | 避免同一 session 同時執行多個操作，造成頁面狀態混亂。 |
| 錯誤截圖 | Playwright screenshot | 自動化失敗時可快速定位頁面狀態。 |
| 操作紀錄 | Audit log | 記錄誰在何時執行何種操作。 |
| Session timeout | 內建排程或 cron | 閒置 session 自動釋放，避免資源耗盡。 |
| 結構化錯誤碼 | 自訂 error code | 前端可依錯誤類型顯示對應訊息。 |

### 16-2 後期補強

| 補強項目 | 建議技術 | 適用情境 |
|----------|----------|----------|
| Session metadata 持久化 | Redis | 服務重啟後仍可保留 session metadata，但 Playwright context 本身仍需重建。 |
| Playwright trace | Trace viewer | 複雜操作失敗時可回放流程。 |
| 操作結果推播 | WebSocket / SSE | 長時間操作可即時回報進度。 |
| 多智能體協作 | OpenCode / Agents SDK | 更複雜的多步驟任務規劃，屬後期規劃。 |

## 十七、重要風險與對策

### 17-1 AI 誤操作風險

AI 不應直接產生 selector、URL 或 Playwright code。此風險主要發生於後期導入 AI Resolver 後；第一版不導入 AI Resolver，因此不應存在 AI 直接操作瀏覽器的流程。

錯誤方向：

```text
AI 回傳：點擊第三個按鈕
Playwright 直接照做
```

正確方向：

```text
AI 回傳：intent = query_report
系統確認 query_report 已註冊
再由 queryReport() 執行固定流程
```

### 17-2 同一 session 併發操作風險

同一個 Playwright page 不應同時執行兩個操作。

建議：

```text
READY 才可接收指令
執行中改為 BUSY
完成後改回 READY
失敗則改為 ERROR 或 EXPIRED
```

後期應加入 Queue，例如 BullMQ，避免多指令同時進入同一 session。

### 17-3 舊 PHP 系統頁面狀態風險

`login.php` / `menu.php` 可能存在：

```text
跳轉不固定
alert 彈窗
iframe / frame
登入逾時
重複登入
表單欄位 name 不固定
慢速頁面載入
```

因此 Playwright 操作應該要：

```text
等待 URL
等待關鍵文字
等待 selector
必要時截圖
錯誤時記錄目前 URL 與頁面內容
```

### 17-4 帳密安全風險

因前端輸入的是 `10.10.10.1` 的帳密，務必避免保存與外洩。

建議：

```text
登入 API 全程使用 HTTPS 或內網安全通道
帳密不進 log
錯誤訊息不回傳帳密
Session Manager 不長期保存 password
前端只保存 appSessionId
```

### 17-5 Session 被錯用風險

不可只靠前端傳入的 `userId` 決定 session 歸屬。

建議：

```text
session.userId 必須等於 token.userId
appSessionId 必須由後端產生
前端不可自行指定 sessionId
每次 command 都要驗證 session ownership
```

```text

暫不納入第一版：

Redis session metadata
BullMQ Queue
WebSocket / SSE
多智能體協作
```

原因是第一版應先確認 Playwright 對目標站台的可操作性與穩定性。

## 二十一、結論

本整併版的核心結論：

```text
目前階段應先讓 Playwright 能穩定登入與操作。
每個功能都有固定函式是最終目標，但不需要一開始完成。
舊版 AI 即期導入規劃已廢除。
AI Resolver 改為後期導入。
應採「先探索、再封裝、再註冊、最後接 AI」的漸進式路線。
```

最終方向：

```text
MyAppDesign 前端輸入 10.10.10.1 帳密
  ↓
EasyAPI 驗核
  ↓
Session Manager 建立 Playwright session
  ↓
登入 login.php 並停在 menu.php
  ↓
初期用探索函式操作
  ↓
穩定後封裝成 browser.actions.ts
  ↓
累積後建立 action.registry.ts
  ↓
最後導入 AI Resolver
```

這樣可以降低早期複雜度，也能避免 AI、Session、Playwright 三個問題同時混在一起除錯。
