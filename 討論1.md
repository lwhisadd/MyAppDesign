# 前台技術棧選型討論 — 第一次

> 討論日期：2026-06-14  
> 專案：MyAppDesign  
> 議題：前台技術棧 `Ant Design + TypeScript + Node.js + Next.js` 的評估與補強  
> 更新：移除 Tabulator，統一以 Ant Design Table 作為表格方案

---

## 一、確認技術棧

| 層次 | 技術 | 角色 |
|------|------|------|
| 前端框架 | Next.js | App Router / SSR / SSG / 路由 |
| 語言 | TypeScript | 靜態型別 |
| UI 元件庫 | Ant Design | 主 UI 系統、表單、表格、版型 |
| 執行環境 / 後端 | Node.js | API runtime / BFF |

> **Tabulator 已移除**：Ant Design 內建 Table 元件已支援固定欄、Sticky、Virtual Scroll、Sort/Filter、Row Selection，可滿足絕大多數後台需求，不需引入額外表格引擎。

---

## 二、主要問題與風險

### 2-1 SSR / Hydration 邊界

Ant Design 在 Next.js App Router 下需要 `@ant-design/nextjs-registry` 處理首屏樣式注入，否則會出現樣式閃爍（FOUC）。應在 `app/layout.tsx` 的最外層包覆 `AntdRegistry`。

### 2-2 缺少資料層（Data Fetching Layer）

目前技術清單缺乏「資料抓取 / 快取 / 同步」方案。若沒有統一的 server state 管理，分頁、篩選、背景刷新、Optimistic Update 等功能會分散在各組件，難以複用與測試。

### 2-3 Node.js 角色模糊

Node.js 若與 Next.js 同 repo，需明確定義其角色：
- **Next.js Route Handlers**（App Router 內建）
- **獨立 Express / Fastify 服務**（另一個 port / process）
- **BFF（Backend for Frontend）**

三者責任劃分不同，應在架構文件中明確記錄。

### 2-4 打包效能

Ant Design 元件數量多，若不做 tree-shaking，首屏 JS 體積偏重。建議搭配 `@ant-design/nextjs-registry` 與 Next.js 內建的 import optimization 減少無用程式碼。

## 三、補強建議

### 必補

| 補強項目 | 建議技術 | 理由 |
|----------|----------|------|
| 資料抓取與快取 | `TanStack Query (React Query)` | server state 統一管理，處理 loading/error/refetch/pagination |
| Ant Design SSR 樣式 | `@ant-design/nextjs-registry` | 避免 App Router 首屏樣式閃爍 |
| Runtime 型別驗證 | `Zod` | API input/output schema，補齊只有 TS interface 沒有 runtime guard 的缺口 |

### 視需求補

| 補強項目 | 建議技術 | 適用情境 |
|----------|----------|----------|
| 複雜表單管理 | `React Hook Form` | 多步驟、複雜驗證、可測試封裝 |
| 大型表格效能 | Ant Table `virtual` 屬性 或 `@tanstack/react-virtual` | 超過 1 萬筆 row 的 virtual scroll |

## 四、最終確認技術棧

```
前端框架        Next.js (App Router) + TypeScript
UI 系統         Ant Design（含 Ant Table）
資料層          TanStack Query
表單層          Ant Form（一般）/ React Hook Form（複雜）

```

## 六、結論

移除 Tabulator 後，技術棧更簡潔、一致性更高：

1. **表格統一走 Ant Table**，維護單一 API 與樣式體系，與 Ant Design 表單、Modal、Drawer 無縫整合。
2. **缺少資料層（TanStack Query）是功能性缺口**，越早補越容易，越晚補重構成本越高。
3. **SSR 邊界與 Ant Design 樣式注入是 Next.js App Router 的已知地雷**，需要在專案初期就處理。

> 下一步建議：決定 Node.js 的角色（Route Handlers vs 獨立服務），並建立 `AppTable` wrapper 統一表格抽象層。
