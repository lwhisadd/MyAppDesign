| 層次 | 技術 | 角色 |
|------|------|------|

## 二、主要問題與風險

### 2-1 Ant Design 與 Tabulator 責任重疊

Ant Design 本身已內建完整的 Table 元件，支援固定欄、Sticky、Virtual Scroll、Sort/Filter、Row Selection 等功能。Tabulator 也是一套完整的表格引擎，擁有自己的 DOM 管理、事件模型、樣式體系。兩者同時存在同一個前台時，會產生：

- **設計不一致**：兩套 hover 色、border、spacing 標準
- **維護成本加倍**：要理解兩套 API 與事件模型
- **樣式覆蓋衝突**：Tabulator 的 CSS class 與 Ant Design 的 CSS-in-JS token 難以共存

**結論：兩者不應平等存在，必須主次明確。**

### 2-2 SSR / Hydration 邊界

Tabulator 屬於高度依賴 DOM 的 library（window、document 操作），在 Next.js 的 Server Components 或 SSR 環境下會觸發 hydration mismatch。

需要使用：
```tsx
const TabulatorTable = dynamic(() => import('@/components/TabulatorTable'), { ssr: false });
```

Ant Design 在 Next.js App Router 下也需要 `@ant-design/nextjs-registry` 處理首屏樣式注入，否則會出現樣式閃爍（FOUC）。

### 2-3 缺少資料層（Data Fetching Layer）

目前技術清單缺乏「資料抓取 / 快取 / 同步」方案。若沒有統一的 server state 管理，分頁、篩選、背景刷新、Optimistic Update 等功能會分散在各組件，難以複用與測試。

### 2-4 Node.js 角色模糊

Node.js 若與 Next.js 同 repo，需明確定義其角色：
- **Next.js Route Handlers**（App Router 內建）
- **獨立 Express / Fastify 服務**（另一個 port / process）
- **BFF（Backend for Frontend）**

三者責任劃分不同，應在架構文件中明確記錄。

### 2-5 打包效能

Ant Design + Tabulator 的 bundle 相加較重。若不做 tree-shaking 與動態載入，首屏 JS 體積會超標。

## 三、補強建議

### 必補

| 補強項目 | 建議技術 | 理由 |
|----------|----------|------|
| 資料抓取與快取 | `TanStack Query (React Query)` | server state 統一管理，處理 loading/error/refetch/pagination |
| SSR 邊界處理 | `next/dynamic({ ssr: false })` | Tabulator 等 DOM-first 元件的 client-only island |
| Ant Design SSR 樣式 | `@ant-design/nextjs-registry` | 避免 App Router 首屏樣式閃爍 |
| Runtime 型別驗證 | `Zod` | API input/output schema，補齊只有 TS interface 沒有 runtime guard 的缺口 |

### 視需求補

| 補強項目 | 建議技術 | 適用情境 |
|----------|----------|----------|
| 複雜表單管理 | `React Hook Form` | 多步驟、複雜驗證、可測試封裝 |
| 設計 Token 統一 | Ant Design `theme.token` | 讓 Tabulator 的 CSS variables 對齊 Ant Design 顏色體系 |
| 大型表格效能 | `@tanstack/react-virtual` | 超過 1 萬筆 row 的 virtual scroll（Ant Table 內建 virtual 也可考慮） |

## 四、Tabulator vs Ant Table 決策準則

| 條件 | 建議 |
|------|------|
| CRUD 後台、一般報表、分頁篩選 | **Ant Table 就夠**，不需要 Tabulator |
| 需要 inline cell editing（像 Excel） | 考慮 Tabulator |
| 超大資料量（> 50K rows）且需要複雜互動 | 考慮 Tabulator（client-only） |
| 需要與 Ant Design 表單、Modal、Drawer 緊密整合 | 用 Ant Table，整合成本最低 |

**建議策略：預設用 Ant Table，遇到 Tabulator 才能解決的需求時，以 `dynamic({ ssr: false })` 封裝成獨立 client island，不要混入主 UI 流。**

```
資料層          TanStack Query

```

## 七、結論

這次討論最核心的結論有三點：

1. **Tabulator 與 Ant Design 的表格能力重疊是最大的架構風險**，應明確降 Tabulator 為特例工具，不做全域引入。
2. **缺少資料層（TanStack Query）是功能性缺口**，越早補越容易，越晚補重構成本越高。
3. **SSR 邊界與 Ant Design 樣式注入是 Next.js App Router 的已知地雷**，需要在專案初期就處理，而不是遇到 bug 再修。

> 下一步建議：決定 Node.js 的角色（Route Handlers vs 獨立服務），並建立 `AppTable` wrapper 統一表格抽象層。
