## 二、修正後的登入流程

原始方向是「MyAppDesign 登入後，再由後端取得或傳遞登入資訊」。後續確認後，實際情境應修正為：

```text
使用者在 MyAppDesign 前端輸入 10.10.10.1 的帳號密碼
  ↓
MyAppDesign 後端 API 接收帳密
  ↓
後端呼叫 EasyAPI 驗核身份
  ↓
驗核成功後，後端將同一組帳密短暫交給 Session Manager
  ↓
Session Manager 使用 Playwright 登入 http://10.10.10.1/login.php
  ↓
成功進入 menu.php
  ↓
建立該使用者專屬 Playwright Session
  ↓
MyAppDesign 回傳 appSessionId 給前端
```

因此，A 與 B 目標應修正如下：

| 項目 | 修正版說明 |
|------|------------|
| A | 使用者於 MyAppDesign 前端輸入 `10.10.10.1` 的帳號密碼，送至後端 API，由 EasyAPI 驗核身份。 |
| B | 驗證成功後，後端將該帳密短暫交給 Session Manager，由 Playwright 建立專用 BrowserContext，登入 `http://10.10.10.1/login.php`，成功後進入 `menu.php`。 |

## 三、帳密處理原則

雖然前端登入頁輸入的是 `10.10.10.1` 的帳密，但系統仍應避免長期保存敏感資料。

### 3-1 前端原則

前端可以輸入帳密，但不可保存帳密：

```text
不可存 localStorage
不可存 sessionStorage
不可寫入 cookie
不可在 console.log 印出
不可在錯誤訊息中回傳
不可在 URL query string 中傳遞
```

前端登入後，應只保存 MyAppDesign 自己發出的 `appSessionId` 或應用層 session token。

### 3-2 後端原則

後端可以在登入流程中短暫使用帳密：

```text
1. 接收 username / password
2. 呼叫 EasyAPI 驗核
3. 驗核成功後交給 Session Manager 建立 Playwright session
4. 登入完成後即不再保存 password
```

Session Manager 內部 session 不應長期保存明文密碼。

建議 session 資料只保留：

```ts
type ManagedSession = {
  sessionId: string
  userId: string
  context: BrowserContext
  page: Page
  state: SessionState
  createdAt: Date
  lastActiveAt: Date
  currentUrl?: string
}
```

不建議長期保留：

```ts
{
  username: string,
  password: string
}
```

若未來需要自動重新登入，可再評估安全保存方式；第一階段建議先採「session 失效後要求使用者重新登入」。

## 四、現階段開發策略：Playwright 先行

目前最重要的目標不是一次完成完整 AI Agent 或完整 `browser.actions.ts`，而是先確認 Playwright 可以穩定操作目標站台。

第一階段目標：

```text
1. 可以啟動 Chromium
2. 可以建立 BrowserContext
3. 可以開啟 http://10.10.10.1/login.php
4. 可以填入使用者帳密
5. 可以成功登入
6. 可以進入 menu.php
7. 可以在 menu.php 操作至少一個功能
8. 可以回傳操作結果或錯誤原因
```

此階段建議使用 `headless: false`，方便觀察實際頁面行為。

```ts
import { chromium } from 'playwright'

async function main() {
  const browser = await chromium.launch({ headless: false })
  const context = await browser.newContext()
  const page = await context.newPage()

  await page.goto('http://10.10.10.1/login.php')

  await page.fill('#username', process.env.TEST_USERNAME!)
  await page.fill('#password', process.env.TEST_PASSWORD!)
  await page.click('button[type="submit"]')

  await page.waitForURL('**/menu.php')

  console.log('login success:', page.url())

  await page.pause()
}

main()
```

> 注意：上方 selector 僅為示意，實際需依 `login.php` HTML 欄位名稱調整。

## 五、browser.actions.ts 的定位修正

`browser.actions.ts` 是最終目標，但不應在第一階段就要求完整設計所有功能。

較合理的定位是：

```text
browser.actions.ts = 已驗證成功的 Playwright 操作函式集合
```

而不是：

```text
browser.actions.ts = 預先設計完整的所有功能清單
```

也就是說，開發方式應採：

```text
先用 Playwright 直接測試操作
  ↓
確認某個流程能穩定執行
  ↓
將該流程封裝成函式
  ↓
放入 browser.actions.ts
  ↓
逐步形成固定函式庫
```

## 六、建議採用「探索函式 → 固定函式」模式

### 6-1 初期：探索函式

初期可以先建立較彈性的探索函式，協助確認 menu.php 的操作流程。

```ts
import type { Page } from 'playwright'

export async function goMenu(page: Page) {
  await page.goto('http://10.10.10.1/menu.php', {
    waitUntil: 'domcontentloaded'
  })

  return {
    ok: true,
    action: 'goMenu',
    url: page.url()
  }
}

export async function clickMenuItem(page: Page, text: string) {
  await page.click(`text=${text}`)

  await page.waitForLoadState('domcontentloaded').catch(() => {})

  return {
    ok: true,
    action: 'clickMenuItem',
    text,
    url: page.url()
  }
}
```

這類函式適合早期探索：

```text
點選選單
觀察跳轉
確認 frame / iframe
確認表單欄位
確認查詢按鈕
確認結果表格
```

### 6-2 穩定後：固定函式

當某個功能流程確認穩定後，再封裝成正式函式。

例如：

```ts
import type { Page } from 'playwright'

export async function queryReport(
  page: Page,
  params: {
    year: number
    month: number
  }
) {
  await page.click('text=報表查詢')
  await page.fill('#year', String(params.year))
  await page.fill('#month', String(params.month))
  await page.click('text=查詢')

  await page.waitForLoadState('networkidle').catch(() => {})

  return {
    ok: true,
    action: 'queryReport',
    params,
    url: page.url()
  }
}
```

此時 `queryReport()` 才是可供後續 AI Resolver 或 Command Registry 使用的正式 action。

## 八、browser.login.ts 建議先獨立

登入是所有操作的基礎，建議先獨立為 `browser.login.ts`。

```ts
import type { Page } from 'playwright'

export async function loginToTargetSite(
  page: Page,
  credentials: {
    username: string
    password: string
  }
) {
  await page.goto('http://10.10.10.1/login.php', {
    waitUntil: 'domcontentloaded'
  })

  await page.fill('#username', credentials.username)
  await page.fill('#password', credentials.password)

  await Promise.all([
    page.waitForNavigation({ waitUntil: 'domcontentloaded' }),
    page.click('button[type="submit"]')
  ])

  if (!page.url().includes('menu.php')) {
    throw new Error(`LOGIN_FAILED_CURRENT_URL=${page.url()}`)
  }

  return {
    ok: true,
    url: page.url()
  }
}
```

正式實作時建議不要只檢查 URL，也要檢查登入後頁面是否真的出現 menu 內容，例如：

```ts
const hasMenu = await page.locator('text=主選單').count()
const hasLogout = await page.locator('text=登出').count()

if (!hasMenu && !hasLogout) {
  throw new Error('LOGIN_FAILED_MENU_NOT_READY')
}
```

## 九、Session Manager 初期設計

初期可先使用記憶體 Map 管理 session。

```ts
type SessionState =
  | 'CREATING'
  | 'LOGGING_IN'
  | 'READY'
  | 'BUSY'
  | 'EXPIRED'
  | 'ERROR'
  | 'CLOSED'

type ManagedSession = {
  sessionId: string
  userId: string
  context: BrowserContext
  page: Page
  state: SessionState
  createdAt: Date
  lastActiveAt: Date
  currentUrl?: string
}
```

操作流程：

```text
取得 session
  ↓
確認 session.state === READY
  ↓
將狀態改為 BUSY
  ↓
執行 Playwright 操作
  ↓
更新 lastActiveAt
  ↓
將狀態改回 READY
```

若操作失敗：

```text
記錄錯誤
截圖
將狀態改為 ERROR 或 EXPIRED
必要時關閉 context
回傳明確錯誤碼
```

## 十、API 建議

初期 API 不需要太複雜，可先提供：

```http
POST /api/login
POST /api/command
GET /api/session/:sessionId/status
POST /api/session/:sessionId/close
```

### 10-1 登入 API

```http
POST /api/login
```

Request：

```json
{
  "username": "user001",
  "password": "******"
}
```

後端流程：

```text
1. 呼叫 EasyAPI 驗核帳密
2. 驗核失敗則回傳 401
3. 驗核成功則建立 Playwright BrowserContext
4. 使用同一組帳密登入 login.php
5. 成功進入 menu.php
6. 回傳 appSessionId
```

Response：

```json
{
  "ok": true,
  "appSessionId": "sess_xxx",
  "user": {
    "id": "user001",
    "name": "王小明"
  }
}
```

### 10-2 指令 API

第一階段建議先接受結構化指令，不直接接自然語言。

```http
POST /api/command
```

Request：

```json
{
  "appSessionId": "sess_xxx",
  "intent": "click_menu_item",
  "params": {
    "text": "報表查詢"
  }
}
```

待操作流程穩定後，再接 AI Resolver。

## 十一、AI Resolver 應延後導入

AI Resolver 的最終目標是：

```text
自然語言 → 結構化 command
```

例如：

```text
使用者：幫我查 2026 年 6 月報表
```

轉成：

```json
{
  "intent": "query_report",
  "params": {
    "year": 2026,
    "month": 6
  }
}
```

但在 Playwright 操作尚未穩定之前，不建議先導入 AI。

原因：

```text
AI 可能解析正確，但瀏覽器操作尚未穩定
瀏覽器操作錯誤時，難以判斷是 AI 問題還是 Playwright 問題
早期除錯成本會過高
```

建議導入順序：

```text
1. Playwright 登入成功
2. menu.php 基本操作成功
3. 第一個固定 action 成功
4. 多 session 管理穩定
5. 再加入 AI Resolver
```

## 十二、Action Registry 後期再加入

當 `browser.actions.ts` 累積多個穩定函式後，再建立 `action.registry.ts`。

```ts
import { queryReport } from './browser.actions'

export const actionRegistry = {
  query_report: queryReport
}
```

執行時：

```ts
const action = actionRegistry[command.intent]

if (!action) {
  throw new Error(`UNKNOWN_ACTION: ${command.intent}`)
}

const result = await action(session.page, command.params)
```

Action Registry 的作用是：

```text
限制 AI 或前端只能呼叫已允許的功能
避免任意操作瀏覽器
讓每個功能都有固定入口
方便測試與權限控管
```

## 十四、重要風險與對策

### 14-1 AI 誤操作風險

AI 不應直接產生 selector、URL 或 Playwright code。

錯誤方向：

```text
AI 回傳：點擊第三個按鈕
Playwright 直接照做
```

正確方向：

```text
AI 回傳：intent = query_report
系統確認 query_report 已註冊
再由 queryReport() 執行固定流程
```

### 14-2 同一 session 併發操作風險

同一個 Playwright page 不應同時執行兩個操作。

建議：

```text
READY 才可接收指令
執行中改為 BUSY
完成後改回 READY
失敗則改為 ERROR 或 EXPIRED
```

後期應加入 Queue，例如 BullMQ，避免多指令同時進入同一 session。

### 14-3 舊 PHP 系統頁面狀態風險

`login.php` / `menu.php` 可能存在：

```text
跳轉不固定
alert 彈窗
iframe / frame
登入逾時
重複登入
表單欄位 name 不固定
慢速頁面載入
```

因此 Playwright 操作應該要：

```text
等待 URL
等待關鍵文字
等待 selector
必要時截圖
錯誤時記錄目前 URL 與頁面內容
```

### 14-4 帳密安全風險

因前端輸入的是 `10.10.10.1` 的帳密，務必避免保存與外洩。

建議：

```text
登入 API 全程使用 HTTPS 或內網安全通道
帳密不進 log
錯誤訊息不回傳帳密
Session Manager 不長期保存 password
前端只保存 appSessionId
```

## 十五、結論

本補充版的核心結論是：

```text
目前階段應先讓 Playwright 能穩定登入與操作。
每個功能都有固定函式是最終目標，但不需要一開始完成。
應採「先探索、再封裝、再註冊、最後接 AI」的漸進式路線。
```

最終建議架構：

```text
[MyAppDesign 前端]
  使用者輸入 10.10.10.1 帳密
        ↓
[MyAppDesign 後端 API]
  EasyAPI 驗核帳密
        ↓
[Session Manager 10.10.10.1:3002]
  建立 BrowserContext
  Playwright 登入 login.php
  進入 menu.php
        ↓
[Playwright 探索操作]
  先跑通流程
        ↓
[browser.actions.ts]
  將穩定流程逐步封裝成固定函式
        ↓
[action.registry.ts]
  只允許已註冊 action
        ↓
[AI Resolver]
  自然語言轉成 intent + params
```

因此，現階段最合理的開發目標是：

```text
1. 完成 Playwright 登入測試
2. 完成 menu.php 基礎探索操作
3. 完成 Session Manager 基礎版
4. 將第一個穩定功能封裝進 browser.actions.ts
5. 再逐步擴充其他功能
```

這樣可以降低早期複雜度，也能避免 AI、Session、Playwright 三個問題同時混在一起除錯。
