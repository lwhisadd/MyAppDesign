> 討論日期：2026-06-29  
> 專案：MyAppDesign  
> 議題：瀏覽器自動化 + 智能體語意解析 + 多 Session 管理 + 檔案型回傳 + 連續自然語言操作

## 一、文件目的

本文件整併前次討論內容，並依據最新需求修正版本策略。

最新確認的核心要求是：**每一個版本都必須完成自然語言操作閉環**。

也就是說，每一版都不能只做到 Playwright 登入或手動 command 操作，而是至少必須支援：

```text
使用者登入
  ↓
使用者在瀏覽器操作介面輸入自然語言
  ↓
智能體解析自然語言
  ↓
Playwright 操作 menu.php 與後續多層頁面
  ↓
取得答案
  ↓
將答案整理成檔案
  ↓
回傳到使用者操作介面
  ↓
使用者可針對該檔案繼續自然語言操作，或結束對話
```

因此，舊版中「AI Resolver 後期才導入、第一版完全不做 AI」的規劃已不再適用。

---

## 二、版本決策聲明

本文件廢除以下舊版規劃：

```text
第一版只做 Playwright 登入與 menu.php 探索
第一版不導入 AI Resolver
第一版只接受結構化 command
AI Resolver 留到後期才導入
```

修正後的版本決策如下：

1. **每一版都必須支援自然語言輸入。**
2. **每一版都必須具備智能體語意解析能力。**
3. **每一版都必須能透過 Playwright 操作 `menu.php` 與後續功能頁面。**
4. **每一版都必須能產生答案檔案並回傳使用者操作介面。**
5. **每一版都必須支援使用者針對回傳檔案進行下一步自然語言操作，或結束對話。**
6. 不同版本的差異不在於「有沒有 AI」，而在於：
   - 支援的自然語言範圍
   - 支援的功能頁面數量
   - 可處理的檔案格式
   - 多步驟任務能力
   - 錯誤恢復能力
   - Session 管理成熟度

因此，本文件後續提到「第一版」時，指的是：

```text
具備最小可用自然語言智能體閉環的版本
```

而不是只具備 Playwright 登入測試的技術原型。

## 三、每一版本必須完成的共同目標

每一個版本都必須完成以下四項目標。

### 3-1 使用者登入後可輸入自然語言

使用者完成登入後，必須能在 MyAppDesign 操作介面上輸入自然語言，例如：

```text
上半年應收帳款總計
```

或：

```text
查詢今年 1 月到 6 月的應收帳款總額，整理成報表
```

這代表前端不能只提供固定按鈕或結構化表單，至少要提供自然語言輸入框。

### 3-2 自然語言須經智能體解析並操作 menu.php

自然語言指令必須透過智能體解析，例如：

```text
OpenCode
Hermes
Ollama + Hermes
OpenAI API
其他可替代 LLM / Agent Runtime
```

智能體解析後，應產生結構化任務計畫，再由 Session Manager 與 Playwright 執行。

例如：

```json
{
  "intent": "query_accounts_receivable_summary",
  "params": {
    "period": "first_half",
    "year": 2026
  },
  "steps": [
    {
      "action": "navigate_menu",
      "target": "accounts_receivable"
    },
    {
      "action": "open_function",
      "target": "summary_report"
    },
    {
      "action": "set_period",
      "from": "2026-01-01",
      "to": "2026-06-30"
    },
    {
      "action": "submit_query"
    },
    {
      "action": "extract_result"
    }
  ]
}
```

Playwright 操作不一定只停在 `menu.php`，可能會經過多層頁面：

```text
menu.php
  ↓
function_1.php
  ↓
function_1_sub_1.php
  ↓
查詢結果頁
```

因此，每一版都要具備基本的多頁面導航能力。

### 3-3 答案須以檔案型式回傳使用者操作介面

查詢或操作完成後，系統不能只回傳純文字訊息，必須產生一個可供下載、預覽或後續操作的檔案。

第一版可先支援簡單格式：

```text
.md
.json
.csv
```

後續版本再擴充：

```text
.xlsx
.pdf
.docx
zip
```

例如「上半年應收帳款總計」可產生：

```text
accounts_receivable_2026_H1_summary.md
accounts_receivable_2026_H1_summary.json
accounts_receivable_2026_H1_summary.csv
```

檔案內容至少應包含：

```text
使用者原始自然語言指令
智能體解析後的 intent / params
操作路徑
查詢條件
結果摘要
原始擷取資料
產生時間
sessionId / requestId
```

### 3-4 使用者可針對檔案繼續自然語言操作或結束

檔案回傳後，使用者可以繼續提出自然語言指令，例如：

```text
把這份檔案改成 Excel
```

```text
依客戶別重新分組
```

```text
把金額超過 10 萬的項目列出來
```

```text
產生主管摘要
```

```text
結束此對話
```

因此，系統需要保留對話上下文與檔案上下文：

```text
conversationId
appSessionId
lastArtifactId
artifact metadata
原始查詢結果
後續操作紀錄
```

## 四、修正後的整體架構

```text
[MyAppDesign 前端]
  使用者輸入 10.10.10.1 帳密
        ↓
[MyAppDesign 後端 API]
  EasyAPI 驗核帳密
        ↓
[Session Manager 10.10.10.1:3002]
  建立 Playwright BrowserContext
  登入 login.php
  成功後進入 menu.php
        ↓
[自然語言操作介面]
  使用者輸入自然語言指令
        ↓
[Agent Resolver]
  Hermes / OpenCode / OpenAI API / 其他 Agent
  將自然語言解析為 Task Plan
        ↓
[Command Validator]
  驗證 intent、params、allowed actions
        ↓
[Action Executor]
  呼叫 browser.actions.ts
        ↓
[Playwright 多頁面操作]
  menu.php → function_1.php → function_1_sub_1.php → result
        ↓
[Result Extractor]
  擷取表格、文字、金額、狀態
        ↓
[Artifact Generator]
  產生 md / json / csv / xlsx / pdf 等檔案
        ↓
[Artifact Store]
  保存檔案與 metadata
        ↓
[MyAppDesign 前端]
  回傳檔案給使用者
        ↓
[Follow-up Agent]
  使用者可針對檔案繼續自然語言操作或結束
```

核心原則：

```text
每一版都要有自然語言智能體閉環
AI 可以做語意解析與任務規劃
AI 不直接自由操作瀏覽器
Playwright 只執行受控 action
所有 action 必須經過 Command Validator
所有結果必須產生檔案 Artifact
檔案必須可成為下一輪自然語言操作的上下文
```

## 五、登入與 Session 流程

使用者登入 MyAppDesign 時，輸入的是 `10.10.10.1` 站台的帳號密碼。

```text
使用者在 MyAppDesign 前端輸入 10.10.10.1 的帳號密碼
  ↓
MyAppDesign 後端 API 接收 username / password
  ↓
後端呼叫 EasyAPI 驗核身份
  ↓
驗核成功後，後端將同一組帳密短暫交給 Session Manager
  ↓
Session Manager 使用 Playwright 開啟 http://10.10.10.1/login.php
  ↓
Playwright 填入帳密並送出登入表單
  ↓
登入成功後進入 menu.php
  ↓
建立該使用者專屬 Playwright Session
  ↓
MyAppDesign 回傳 appSessionId / conversationId 給前端
```

前端後續所有操作都使用：

```text
appSessionId
conversationId
```

不再重複傳送密碼。

## 六、帳密處理原則

前端可以讓使用者輸入 `10.10.10.1` 的帳密，但不可保存帳密。

不可執行：

```text
不可存 localStorage
不可存 sessionStorage
不可寫入 cookie
不可在 console.log 印出
不可在錯誤訊息中回傳
不可在 URL query string 中傳遞
```

後端可以在登入流程中短暫使用帳密：

```text
1. 接收 username / password
2. 呼叫 EasyAPI 驗核
3. 驗核成功後交給 Session Manager 建立 Playwright session
4. Playwright 登入完成後即不再保存 password
```

Session Manager 內部 session 不應長期保存明文密碼。

建議 session 只保留：

```ts
type ManagedSession = {
  sessionId: string
  userId: string
  conversationId: string
  context: BrowserContext
  page: Page
  state: SessionState
  createdAt: Date
  lastActiveAt: Date
  currentUrl?: string
  lastArtifactId?: string
}
```

## 七、智能體語意解析設計

每一版都必須具備 Agent Resolver。

Agent Resolver 的任務不是直接操作瀏覽器，而是把自然語言轉成受控任務計畫。

### 7-1 輸入

```json
{
  "appSessionId": "sess_xxx",
  "conversationId": "conv_xxx",
  "naturalLanguage": "上半年應收帳款總計",
  "lastArtifactId": null
}
```

### 7-2 輸出

```ts
type AgentPlan = {
  intent: string
  confidence: number
  params: Record<string, string | number | boolean>
  requiredActions: string[]
  steps: AgentStep[]
  expectedArtifact: {
    type: 'md' | 'json' | 'csv' | 'xlsx' | 'pdf'
    title: string
  }
  missingFields?: string[]
}

type AgentStep = {
  action: string
  target?: string
  params?: Record<string, string | number | boolean>
}
```

### 7-3 信心不足處理

若智能體信心不足或缺少必要條件，不應直接操作瀏覽器。

```ts
if (plan.confidence < 0.75 || plan.missingFields?.length) {
  return {
    status: 'need_confirmation',
    message: '請補充查詢年度或期間。',
    missingFields: plan.missingFields
  }
}
```

## 八、Playwright 多頁面操作設計

Playwright 操作可能經過多重連結。

例如：

```text
menu.php
  ↓
function_1.php
  ↓
function_1_sub_1.php
  ↓
result.php
```

因此 `browser.actions.ts` 不應只支援單一 click，而要逐步建立以下能力：

```text
登入檢查
回到 menu.php
點選主選單
進入功能頁
進入子功能頁
填寫查詢條件
送出查詢
等待結果
擷取表格
擷取文字
擷取錯誤訊息
下載或產生檔案
```

初期可保留探索型 action：

```ts
export async function clickMenuItem(page: Page, text: string) {
  await page.click(`text=${text}`)
  await page.waitForLoadState('domcontentloaded').catch(() => {})

  return {
    ok: true,
    action: 'clickMenuItem',
    text,
    url: page.url()
  }
}
```

但每一版至少要有一組端到端固定流程，例如：

```ts
export async function queryAccountsReceivableSummary(
  page: Page,
  params: {
    fromDate: string
    toDate: string
  }
) {
  await page.goto('http://10.10.10.1/menu.php')
  await page.click('text=應收帳款')
  await page.waitForURL('**/function_1.php')
  await page.click('text=統計查詢')
  await page.waitForURL('**/function_1_sub_1.php')
  await page.fill('#fromDate', params.fromDate)
  await page.fill('#toDate', params.toDate)
  await page.click('text=查詢')
  await page.waitForSelector('#resultTable')

  const total = await page.locator('#totalAmount').innerText()

  return {
    ok: true,
    total,
    url: page.url()
  }
}
```

> 上方 selector 與頁面名稱為示意，實際需依 `10.10.10.1` 站台 HTML 調整。

## 九、Command Validator 與 Action Registry

智能體產生的 action 不可直接執行，必須先經過白名單驗證。

```ts
export const actionRegistry = {
  query_accounts_receivable_summary: queryAccountsReceivableSummary,
  click_menu_item: clickMenuItem
}
```

執行前檢查：

```ts
const action = actionRegistry[plan.intent]

if (!action) {
  throw new Error(`UNKNOWN_ACTION: ${plan.intent}`)
}

const result = await action(session.page, plan.params)
```

原則：

```text
AI 可以提出 intent
AI 可以提出 params
AI 可以提出 steps
系統必須驗證 steps 是否被允許
系統只執行 actionRegistry 內存在的 action
AI 不可任意產生 selector / URL / Playwright code 後直接執行
```

## 十二、Session Manager 初期設計

```ts
type SessionState =
  | 'CREATING'
  | 'LOGGING_IN'
  | 'READY'
  | 'BUSY'
  | 'WAITING_USER_CONFIRMATION'
  | 'EXPIRED'
  | 'ERROR'
  | 'CLOSED'

type ManagedSession = {
  sessionId: string
  userId: string
  conversationId: string
  context: BrowserContext
  page: Page
  state: SessionState
  createdAt: Date
  lastActiveAt: Date
  currentUrl?: string
  lastArtifactId?: string
}
```

操作流程：

```text
取得 session
  ↓
確認 session.state === READY
  ↓
接收自然語言
  ↓
Agent Resolver 產生 plan
  ↓
Command Validator 驗證 plan
  ↓
將 session.state 改為 BUSY
  ↓
Playwright 執行 action
  ↓
Result Extractor 擷取答案
  ↓
Artifact Generator 產生檔案
  ↓
更新 lastArtifactId
  ↓
將 session.state 改回 READY
```

## 十五、版本規劃

### V1：最小自然語言智能體閉環

V1 必須完成：

```text
使用者登入
自然語言輸入框
Agent Resolver 支援至少 1 個 intent
Playwright 操作 menu.php 與至少 1 條功能路徑
取得答案
產生 md/json 檔案
回傳 artifactId 與檔案
支援針對檔案做至少 1 種後續自然語言操作
支援結束對話
```

V1 可先限制：

```text
只支援少數固定自然語句
只支援 1 個站台功能
只支援 md/json/csv
不支援複雜多輪推理
不支援大量併發
```

### V2：擴充功能與檔案操作

```text
支援多個 intent
支援多條 menu.php → function_x.php 操作路徑
```

## 十六、每版驗收標準

每一版都應通過以下驗收：

```text
1. 使用者可成功登入並建立 appSessionId。
2. 使用者可輸入自然語言指令。
3. 智能體可將自然語言解析為 AgentPlan。
4. Command Validator 可驗證 AgentPlan。
5. Playwright 可從 menu.php 操作到目標功能頁。
6. 系統可取得查詢結果。
7. 系統可產生檔案 Artifact。
8. 前端可取得並顯示 / 下載該檔案。
9. 使用者可針對該檔案輸入下一步自然語言操作。
10. 使用者可結束對話。
```

若任一版本缺少第 2、3、7、9 項，則不符合本文件定義的版本要求。

## 十七、重要風險與對策

### 17-1 AI 誤操作風險

AI 不應直接產生 selector、URL 或 Playwright code 後直接執行。

正確方式：

```text
AI 產生 AgentPlan
Command Validator 驗證
Action Registry 對應固定 action
Playwright 執行固定 action
```

### 17-2 多頁面導航風險

舊 PHP 系統可能存在：

```text
跳轉不固定
alert 彈窗
iframe / frame
登入逾時
表單欄位 name 不固定
慢速頁面載入
```

對策：

```text
每一步都檢查 URL / selector / 關鍵文字
失敗時截圖
記錄目前 URL 與頁面內容
必要時回到 menu.php 重新導航
```

### 17-3 檔案上下文風險

若沒有保存 artifact metadata，後續自然語言操作會失去上下文。

對策：

```text
每個 artifact 都要有 artifactId
每個 conversation 都要保存 lastArtifactId
每次 follow-up 都要明確知道操作哪個檔案
```

### 17-4 Session 併發風險

同一個 Playwright page 不應同時執行兩個操作。

對策：

```text
READY 才可接收指令
執行中改為 BUSY
完成後改回 READY
失敗則改為 ERROR 或 EXPIRED
後期加入 Queue
```

### 17-5 帳密安全風險

```text
登入 API 全程使用 HTTPS 或內網安全通道
帳密不進 log
錯誤訊息不回傳帳密
Session Manager 不長期保存 password
前端只保存 appSessionId / conversationId
```

## 十八、目前最小可行版本 MVP

MVP 不再只是 Playwright 技術測試，而是「最小自然語言智能體閉環」。

MVP 必須做到：

```text
1. 使用者在 MyAppDesign 輸入 10.10.10.1 帳密
2. EasyAPI 驗核
3. Session Manager 建立 Playwright session
4. Playwright 登入 login.php 並進入 menu.php
5. 使用者在前端輸入自然語言，例如：上半年應收帳款總計
6. Agent Resolver 將自然語言解析成一個已支援 intent
7. Command Validator 驗證該 intent
8. Playwright 操作 menu.php 與後續至少一條功能路徑
9. Result Extractor 取得答案
10. Artifact Generator 產生 md/json 檔案
11. 前端回傳檔案給使用者
12. 使用者可針對該檔案下一步操作，或結束對話
```

MVP 可暫不納入：

```text
大量 intent
大量站台功能
複雜多輪推理
大量併發
完整 xlsx/pdf
Redis
BullMQ
WebSocket / SSE
```

## 十九、結論

本版最新結論：

```text
每一版都必須完成自然語言 → 智能體解析 → Playwright 操作 → 檔案回傳 → 檔案後續自然語言操作的閉環。
```

因此，AI / Agent Resolver 不再是後期才導入的選項，而是每一版都必須存在的核心能力。

但 AI 的能力可以逐版擴充：

```text
V1：支援 1 個 intent、1 條功能路徑、md/json 檔案
V2：支援多 intent、多檔案格式、基本檔案操作
V3：支援 Queue、Redis、Trace、正式多 Session
V4：支援多智能體與複雜任務規劃
```

最終方向：

```text
MyAppDesign 前端輸入 10.10.10.1 帳密
  ↓
EasyAPI 驗核
  ↓
Session Manager 建立 Playwright session
  ↓
使用者輸入自然語言
  ↓
Agent Resolver 解析成 AgentPlan
  ↓
Command Validator 驗證
  ↓
Action Registry 派發固定 action
  ↓
Playwright 操作 menu.php 與多層功能頁
  ↓
Result Extractor 取得答案
  ↓
Artifact Generator 產生檔案
  ↓
使用者針對檔案繼續自然語言操作或結束對話
```

這樣才能符合「每一個版本均應完成自然語言操作目標」的要求。
