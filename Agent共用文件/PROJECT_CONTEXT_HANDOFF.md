# 專案上下文管理與任務交接規範

> 建議檔名：`docs/PROJECT_CONTEXT_HANDOFF.md`  
> 適用範圍：**僅限目前 Git 專案／工作目錄**  
> 本文件不得用於修改 Codex、作業系統、使用者帳號或其他專案的全域設定。

---

## 1. 文件目的

本文件用來規範 Agent 在目前專案中如何：

1. 管理長時間工作的對話上下文。
2. 在上下文接近上限前建立可驗證的專案狀態紀錄。
3. 在壓縮上下文、建立新對話或更換 Agent 時完成任務交接。
4. 確保後續 Agent 不會只依賴對話摘要，而會重新核對程式碼、Git 狀態、文件與測試結果。
5. 避免任何專案內指示影響 Codex 的全域設定或其他專案。

本文件描述的是**專案級工作流程**，不是全域 Codex 設定。

---

## 2. 強制作用範圍

Agent 必須遵守以下限制：

### 2.1 允許操作的範圍

除非使用者另有明確指示，Agent 只能：

- 讀取與修改目前專案根目錄之下的檔案。
- 讀取目前專案的 Git branch、commit、status、diff 與 log。
- 執行目前專案既有的測試、建置、lint、型別檢查及分析命令。
- 在目前專案中建立或更新交接文件。
- 在目前專案中建立專案級 `AGENTS.md`、`docs/PROJECT_STATE.md` 或決策紀錄。

### 2.2 禁止操作的範圍

Agent 不得因本文件而：

- 修改使用者家目錄下的 Codex 設定。
- 修改 `~/.codex/` 或其他全域 Codex 設定目錄。
- 建立或修改全域 `AGENTS.md`。
- 修改其他 Git repository。
- 修改作業系統、Shell、IDE 或帳號的全域設定。
- 將本專案規則寫入其他專案。
- 安裝全域套件或變更全域環境，除非使用者明確要求。
- 清除或重設使用者其他對話、其他專案或其他 Agent 的狀態。
- 將專案機密、原始碼或內部文件傳送到專案以外的位置。

如果完成任務似乎需要修改全域設定，Agent 必須停止該項修改，說明原因，並提出專案區域內可行的替代方案。

## 3. 規範檔案分工

專案應將規則與動態狀態分開保存。

### 3.1 `AGENTS.md`

用途：保存長期、穩定、每次執行都應遵守的專案規則。

適合存放：

- 專案架構概要。
- 開發與命名規範。
- 測試要求。
- 禁止事項。
- 文件更新要求。
- 本文件與 `PROJECT_STATE.md` 的讀取指示。

不應存放：

- 暫時性進度。
- 某次任務的未完成清單。
- 臨時錯誤訊息。
- 容易快速過期的 Git 狀態。

建議在專案根目錄 `AGENTS.md` 中加入：

```markdown
## 上下文與任務交接

開始工作前，必須閱讀：

- `docs/PROJECT_CONTEXT_HANDOFF.md`
- `docs/PROJECT_STATE.md`（若存在）

進行長時間工作、壓縮上下文、切換對話或交接任務時，必須依照
`docs/PROJECT_CONTEXT_HANDOFF.md` 執行。
```

### 3.2 `docs/PROJECT_CONTEXT_HANDOFF.md`

用途：保存本文件所定義的上下文管理與交接流程。

特性：

- 屬於目前專案。
- 應納入版本控制。
- 不應包含使用者全域設定。
- 除非流程本身改變，否則不需頻繁修改。

### 3.3 `docs/PROJECT_STATE.md`

用途：保存目前專案正在進行中的動態狀態。

至少應記錄：

- 當前目標。
- 工作範圍。
- 已完成項目。
- 未完成項目。
- 技術決策。
- 已否決方案。
- 已知問題與風險。
- 測試結果。
- Git 狀態。
- 下一步。
- 待確認事項。

此檔案是交接的重要輸入，但**不是唯一事實來源**。後續 Agent 必須用實際程式碼、Git 與測試重新驗證。

### 3.4 `docs/decisions/`

重大技術決策建議以 ADR 或獨立 Markdown 文件保存，例如：

```text
docs/decisions/
├── 0001-query-builder-boundary.md
├── 0002-session-isolation.md
└── 0003-error-handling-policy.md
```

`PROJECT_STATE.md` 只保留決策摘要與連結，避免狀態檔過度膨脹。

## 4. Agent 開始工作時的必要流程

每次新對話、新 Agent 或重大階段開始時，Agent 必須依序執行：

1. 確認目前工作目錄及專案根目錄。
2. 閱讀專案根目錄的 `AGENTS.md`。
3. 閱讀 `docs/PROJECT_CONTEXT_HANDOFF.md`。
4. 若存在，閱讀 `docs/PROJECT_STATE.md`。
5. 檢查目前 Git branch。
6. 檢查 `git status`。
7. 檢查最近的 commit。
8. 檢查未提交差異。
9. 核對狀態文件提到的主要檔案是否存在。
10. 核對「已完成工作」是否真的存在於程式碼中。
11. 執行適當的既有測試、lint、型別檢查或建置。
12. 區分：
    - 已確認事實。
    - 文件陳述但尚未驗證的內容。
    - 文件與實際狀態不一致的內容。
    - 尚待確認事項。
13. 完成上述檢查後，才能繼續修改程式碼。

不得因 `PROJECT_STATE.md` 表示某項工作完成，就直接假設其正確。

---

## 5. 事實來源優先順序

遇到資訊不一致時，Agent 應依下列原則判定：

1. 使用者在目前任務中的明確指示。
2. 實際程式碼與設定檔。
3. Git branch、commit、status 與 diff。
4. 可重現的測試、建置與執行結果。
5. 專案級 `AGENTS.md`。
6. 架構文件與決策紀錄。
7. `PROJECT_STATE.md`。
8. 對話中的摘要與記憶。

對話記憶只能作為工作線索，不能取代程式碼、Git 與測試結果。

## 6. 工作期間的上下文管理

### 6.1 日常工作原則

Agent 應：

- 將重要結論寫入專案文件，不只保留在對話中。
- 將重大決策與原因寫入決策紀錄。
- 將進度與下一步更新至 `PROJECT_STATE.md`。
- 在修改完成後記錄實際執行的驗證命令及結果。
- 對不確定內容標示「待確認」，不得將推測寫成事實。
- 避免在對話中累積大量已過期的探索紀錄。

### 6.2 使用 `/status`

若目前 Codex 環境支援 `/status`，可用來檢查：

- 上下文使用狀態。
- 目前工作階段資訊。
- 是否需要準備 checkpoint。

若命令不可用，Agent 仍應在長時間工作或重大階段完成時主動建立 checkpoint。

### 6.3 使用 `/compact`

若仍在進行同一個任務、同一個 Sprint 或同一個技術方向，但上下文過長，可在完成 checkpoint 後使用 `/compact`。

執行 `/compact` 前必須：

1. 更新 `docs/PROJECT_STATE.md`。
2. 記錄已完成與未完成工作。
3. 記錄關鍵技術決策及原因。
4. 記錄測試命令與結果。
5. 記錄 Git branch、HEAD 與未提交變更。
6. 將不確定內容標示為「待確認」。
7. 確認後續 Agent 或壓縮後的對話可依文件恢復工作。

不得將 `/compact` 視為文件與 Git checkpoint 的替代品。

### 6.4 不依賴 `/clear`

本流程不假設 Codex 一定提供 `/clear`。

需要完全乾淨的上下文時，優先採用：

1. 更新 `PROJECT_STATE.md`。
2. 建立適當的 Git checkpoint。
3. 開啟新對話。
4. 依本文件重新執行接手驗證。

---

## 7. 必須建立 Checkpoint 的時機

發生以下任一情況時，Agent 必須更新 `PROJECT_STATE.md`：

- 準備執行 `/compact`。
- 準備建立新對話。
- 準備交由另一個 Agent 接手。
- 完成一個 Sprint、里程碑或主要子任務。
- 從分析階段切換到實作階段。
- 從實作階段切換到驗證或部署階段。
- 準備進行高風險或大範圍修改。
- 發現原規劃有重大錯誤。
- 工作被中斷前。
- 使用者要求整理目前進度時。
- 對話已包含大量錯誤路徑、過期假設或無關內容。
- Git 中存在重要但尚未提交的變更。

---

## 8. Checkpoint 更新程序

建立 checkpoint 時，Agent 必須：

### 8.1 重新檢查實際狀態

至少檢查：

```bash
git branch --show-current
git status --short
git log -1 --oneline
git diff --stat
git diff
```

並依專案實際情況執行：

```bash
# 範例，應使用專案既有命令
npm test
npm run lint
npm run typecheck
npm run build
```

不得在未執行測試時寫成「測試通過」。

### 8.2 更新狀態文件

Agent 必須將資訊分成：

- 已確認。
- 未確認。
- 已完成。
- 部分完成。
- 尚未開始。
- 阻塞。
- 已知風險。

### 8.3 記錄變更位置

對每個重要項目，應盡量記錄：

- 檔案路徑。
- 類別、函式、模組或設定名稱。
- 修改目的。
- 驗證方式。
- 對其他模組的可能影響。

### 8.4 記錄決策原因

不得只寫「採用方案 A」。

至少應記錄：

- 決定內容。
- 選擇原因。
- 曾考慮的替代方案。
- 否決替代方案的原因。
- 已知代價或風險。

```text

本次工作僅限目前專案，不得修改任何 Codex、作業系統、
使用者帳號或其他專案的全域設定。

開始任何修改前，請依序執行：

1. 確認目前專案根目錄。
2. 閱讀根目錄的 AGENTS.md。

4. 閱讀 docs/PROJECT_STATE.md。
5. 檢查目前 Git branch、git status、最近 commit 與未提交差異。
6. 核對 PROJECT_STATE.md 提到的主要檔案與已完成工作。
7. 執行適當的既有測試、lint、型別檢查或建置。
8. 列出：
   - 已確認事實
   - 文件與實際狀態不一致之處
   - 尚待確認事項
   - 建議的下一步

接手確認完成後，再執行以下工作：

【填入下一個工作目標】
```

## 11. 壓縮上下文前的提示

準備執行 `/compact` 前，可使用：

```text
在壓縮目前上下文之前，請更新 docs/PROJECT_STATE.md。

本次更新僅限目前專案，不得修改任何全域 Codex 設定。

請根據目前對話、實際程式碼、Git 狀態、文件與測試結果整理，
不要只依靠對話記憶。

必須包含：

1. 專案目標。
2. 當前任務與工作範圍。
3. 已確認的系統架構與行為。
4. 已完成、進行中及尚未完成的工作。
5. 重要技術決策與原因。
6. 曾考慮但否決的方案。
7. 已知問題與風險。
8. 已執行的測試命令與實際結果。

12. 待確認事項。

不確定內容必須標示為「待確認」，不得自行推測。
```

## 12. 階段切換原則

### 12.1 適合繼續同一對話

以下情況可保留同一對話，必要時使用 `/compact`：

- 仍在處理同一個明確任務。
- 架構方向未改變。
- 只是上下文逐漸變長。
- 尚未進入新的 Sprint 或主要子系統。
- 目前對話中的推論大多仍然有效。

### 12.2 適合建立新對話

以下情況應建立 checkpoint 後開啟新對話：

- 從分析階段進入實作階段。
- 從一個 Sprint 進入下一個 Sprint。
- 開始修改另一個主要子系統。
- 原對話包含大量錯誤方向或過期資訊。
- 需要讓新 Agent 進行獨立驗證。
- 主要目標或技術方向已改變。
- 對話上下文已難以辨別哪些資訊仍然有效。

### 12.3 適合建立平行分支或 Fork

若環境支援 fork，且需要比較不同方案，可：

- 保留原對話作為基準。
- 在 fork 中探索替代方案。
- 不得讓兩個分支同時修改同一組檔案而不進行整合管理。
- 最終採用方案必須寫入決策紀錄及 `PROJECT_STATE.md`。

## 13. Git Checkpoint 原則

上下文交接文件不能取代 Git。

重大階段完成後，若使用者允許建立 commit，應：

1. 確認變更範圍。
2. 執行必要測試。
3. 更新文件。
4. 檢查 diff。
5. 建立意圖明確的 checkpoint commit。

若未獲授權建立 commit，必須在 `PROJECT_STATE.md` 明確記錄：

- 尚未提交。
- 變更涉及哪些檔案。
- 是否包含未完成或不穩定內容。
- 後續接手者應先檢查哪些差異。

Agent 不得自行執行 push、建立 PR、合併或發布，除非使用者明確要求。

---

## 14. 測試與驗證要求

交接前必須避免以下模糊描述：

- 「應該可以運作」
- 「看起來沒問題」
- 「大概已完成」
- 「測試應該會通過」

應改成可驗證敘述：

```text
已執行：npm test
結果：通過，128 tests passed

已執行：npm run typecheck
結果：失敗，src/example.ts:42 有型別錯誤

未執行：整合測試
原因：需要尚未提供的測試資料庫
```

測試失敗不得隱藏；必須記錄失敗原因、影響與建議處理方式。

---

## 15. 防止摘要失真的規則

Agent 建立摘要或交接文件時不得：

- 將推測寫成事實。
- 將部分完成寫成全部完成。
- 將未執行測試寫成測試通過。
- 忽略未提交變更。
- 省略已知的高風險問題。
- 省略曾經失敗且可能再次發生的方案。
- 只記錄「做了什麼」，不記錄「為什麼這樣做」。
- 省略精確檔案路徑與程式碼位置。
- 用過長的對話內容取代結構化狀態。

所有無法驗證的內容必須標示：

```text
狀態：待確認
```

---

## 16. 任務完成條件

Agent 只有在以下條件均符合時，才能宣告任務完成：

- 使用者要求的範圍已完成。
- 實際修改已核對。
- 必要測試已執行，或已明確說明無法執行的原因。
- 相關文件已更新。
- `PROJECT_STATE.md` 已反映最新狀態。
- 未完成事項、風險與限制已揭露。
- Git 工作目錄狀態已說明。
- 未修改任何未獲授權的全域設定。
- 沒有將專案規則套用到其他專案。


## 18. 核心原則

> 對話上下文是暫時工作記憶；程式碼、Git、測試與專案文件才是可驗證的事實來源。

Agent 應遵循以下最小流程：

```text
讀取專案規則
    ↓
核對 Git、程式碼與測試
    ↓
執行當前任務
    ↓
更新 PROJECT_STATE.md
    ↓
```
