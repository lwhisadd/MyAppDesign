安裝與設定

30秒完成安裝，即可在任何 AI Agent 中使用

⚡ 快速安裝 (skills.sh 方式)

  1. 執行安裝指令：
    npx skills@latest add mattpocock/skills
  2. 互動式選擇要安裝的 skills，以及目標 coding agent（Claude Code、Codex 等）
  3. 務必勾選 /setup-matt-pocock-skills
  4. 在 agent 中執行 /setup-matt-pocock-skills，回答設定問題（issue tracker、triage labels、docs 位置）
  5. 完成！開始使用所有 skills

🔌 Claude Code Plugin 方式 (免維護)

  1. 在 Claude Code 內執行： 
    /plugin marketplace add mattpocock/skills
/plugin install mattpocock-skills@mattpocock
  2. 或在 shell 中執行： 
    claude plugin marketplace add mattpocock/skills
claude plugin install mattpocock-skills@mattpocock
  3. 執行 /setup-matt-pocock-skills 進行每個 repo 的設定
安裝方式特色適合對象
skills.sh檔案複製到 repo，可自行修改客製化想 hack 修改 skill 內容的工程師
Claude Code Plugin受管理的 bundle，自動跟隨作者更新想開箱即用、保持最新版的使用者

設計哲學

解決 AI Coding Agent 最常見的四大失敗模式

🎯

問題 #1：Agent 做了錯的東西

用 /grill-me 或 /grill-with-docs 事前對齊需求，讓 agent 先問你大量問題再動工。

🗣️

問題 #2：Agent 輸出過度冗長

建立共享語言（Shared Language）和 CONTEXT.md，讓 agent 用 1 個詞取代 20 個詞。

🔴

問題 #3：程式碼根本不能跑

用 /tdd 實作 red-green-refactor 迴圈，給 agent 一致的測試反饋。

🏗️

問題 #4：程式碼越來越難改

用 /improve-codebase-architecture 定期掃描架構，防止軟體熵失控。

🔄

推薦使用工作流程

依照使用順序，從初始化到完成功能的完整流程

#步驟使用 Skill說明
1初始化 Repo/setup-matt-pocock-skills每個 repo 只需執行一次
2理解情境/路由/ask-matt不確定該用哪個 skill 時先問這個
3對齊需求/grill-with-docs開始任何功能前的必做步驟
4規劃大型工作/wayfinder超過一個 session 的龐大工作
5整理 Issues/triage把積壓的 issues 標記分類
6建立 Spec/to-spec從對話轉換成規格文件
7拆分 Tickets/to-tickets把 spec 拆成可執行的票券
8實作功能/implement驅動 TDD + Code Review
9架構健檢/improve-codebase-architecture每隔幾天執行一次
10交接工作/handoff壓縮 session 給下一個 agent

Engineering Skills

每日程式開發必備工具

setup-matt-pocock-skills

/setup-matt-pocock-skills
👤 User-invoked

每個 repo 執行一次的初始化設定。配置 issue tracker、triage labels 和文件存放位置。

使用步驟
  1. 在 agent 中輸入 /setup-matt-pocock-skills
  2. Agent 問：使用哪個 issue tracker？回答 GitHub / Linear / 本地檔案
  3. Agent 問：triage 用哪些 labels？依你的專案回答
  4. Agent 問：文件存哪？指定目錄（如 docs/）
  5. 設定完成，即可使用所有其他 skills
⚠️ 每個 repo 只需執行一次，不需重複執行

ask-matt

/ask-matt
👤 User-invoked

Skill 的路由器（Router）。描述你想做的事，它會推薦最適合的 skill。不知道用哪個 skill 時的入口點。

使用步驟
  1. 輸入 /ask-matt
  2. 用自然語言描述你的目標，例如：「我想新增一個登入功能」
  3. Agent 分析後推薦對應的 skill 組合
  4. 依推薦執行相應 skill
💡 每次開始新工作時的最佳起點，特別是對 skills 還不熟悉時

grill-with-docs

/grill-with-docs
👤 User-invoked

最核心的 skill。在做任何程式變更前執行。它是一個深度訪談 session，同時幫你建立 CONTEXT.md（共享語言）和 ADRs（架構決策記錄）。

使用步驟
  1. 輸入 /grill-with-docs 並描述你想做的改動
  2. Agent 開始逐一詢問深度問題，解析需求細節
  3. 回答所有問題（勿跳過，每個問題都有意義）
  4. Agent 協助更新 CONTEXT.md 補充術語定義
  5. 遇到重要架構決策時，Agent 幫你建立 ADR 文件
  6. 訪談結束後，需求已完整對齊，可進入下一步
🌟 每次功能開發前必用，是降低溝通誤差最有效的 skill
📄 自動維護 CONTEXT.md，讓 agent 使用更精準的領域術語

triage

/triage
👤 User-invoked

將 Issues 透過狀態機（State Machine）移動到正確的狀態。自動為 issues 套用你在 setup 時定義的 triage labels。

使用步驟
  1. 輸入 /triage
  2. Agent 列出待處理的 issues
  3. 對每個 issue，Agent 詢問應套用哪個 triage label
  4. 確認後 Agent 自動在 issue tracker 套用 label
  5. 重複直到所有 issues 處理完畢
📋 需先完成 /setup-matt-pocock-skills 並定義好 triage labels

to-spec

/to-spec
👤 User-invoked

把當前對話（通常是 grill-with-docs 之後）自動合成為規格文件（Spec），並發布到 issue tracker。不需要再做訪談，直接從現有討論提煉。

使用步驟
  1. 完成 /grill-with-docs 後，輸入 /to-spec
  2. Agent 分析對話紀錄，自動提煉出規格內容
  3. Agent 草擬 spec 並展示給你確認
  4. 確認後，自動發布到 GitHub Issue / Linear / 本地檔案
⚡ 通常緊接在 grill-with-docs 之後執行，無縫從需求轉為規格

to-tickets

/to-tickets
👤 User-invoked

將計劃、spec 或對話拆分成一組「Tracer Bullet Tickets」，每個 ticket 明確宣告其阻擋依賴（blocking edges）。可寫為本地文字檔或在 issue tracker 建立原生阻擋連結。

使用步驟
  1. 有一份 spec 或計劃後，輸入 /to-tickets
  2. Agent 分析內容，拆分為最小可執行單元的 tickets
  3. Agent 標記每個 ticket 的前置依賴關係
  4. 確認 ticket 清單，指定儲存方式（issue tracker 或本地檔案）
  5. Agent 批次建立所有 tickets 並設定 blocking 關係
🧩 Tracer Bullet 概念：每個 ticket 穿透整個技術堆疊（前端到後端）的一小片功能

implement

/implement
👤 User-invoked

實際執行 spec 或 tickets 中描述的工作。在預先約定的接縫（seams）上驅動 /tdd，最後以 /code-review 收尾後再提交 commit。

使用步驟
  1. 準備好 spec 或 tickets 後，輸入 /implement
  2. 指定要實作的 ticket 或 spec（可提供 issue URL 或 spec 內容）
  3. Agent 自動識別程式碼接縫並啟動 TDD 循環
  4. 每個功能點完成後自動觸發 code-review
  5. Review 通過後提交 commit
🤖 這是整合其他多個 skills 的高階 orchestrator skill

wayfinder

/wayfinder
👤 User-invoked

規劃超出單一 agent session 能容納的龐大工作。以「調查 tickets」地圖的形式呈現在 issue tracker 上，逐一解決直到路徑清晰。

使用步驟
  1. 面對大型功能或重構時，輸入 /wayfinder 並描述目標
  2. Agent 建立一組「調查 tickets」（Investigation Tickets）
  3. 每個 session 只處理一個調查 ticket
  4. 完成調查後更新 ticket 狀態，逐步收斂路徑
  5. 所有調查完成後，整體方向明確，可進入正式實作
🗺️ 適用於架構重寫、大型功能重構等需要多次 session 的工作

improve-codebase-architecture

/improve-codebase-architecture
👤 User-invoked

掃描 codebase 找出可深化的設計機會，以視覺化 HTML 報告呈現，再引導你訪談選定的改善點。建議每隔幾天執行一次。

使用步驟
  1. 輸入 /improve-codebase-architecture
  2. Agent 掃描整個 codebase，找出架構問題點
  3. 生成 HTML 報告，視覺化展示問題清單
  4. 從報告中選擇一個改善項目
  5. Agent 進入 grilling 訪談，深度分析選定問題
  6. 輸出改善建議或直接進入實作
🔥 防止「Big Ball of Mud」問題，是維護 code quality 的核心工具
📅 每幾天執行一次，保持架構健康

tdd

/tdd
🤖 Model-invoked

測試驅動開發（TDD）流程。以 red-green-refactor 迴圈一次一個垂直切面（vertical slice）地實作功能或修復 Bug。通常由 /implement 自動觸發。

使用步驟
  1. 可直接輸入 /tdd 並描述要實作的功能
  2. 🔴 RED：Agent 先寫一個會失敗的測試
  3. 🟢 GREEN：Agent 寫最少程式碼讓測試通過
  4. 🔵 REFACTOR：Agent 重構程式碼，測試持續通過
  5. 重複循環直到功能完成
✅ 提供一致的反饋機制，大幅提升 agent 生成程式碼的品質

diagnosing-bugs

/diagnosing-bugs
🤖 Model-invoked

針對難以追查的 Bug 和效能回歸問題的系統化診斷迴圈。流程：重現 → 最小化 → 假設 → 插入測量點 → 修復 → 回歸測試。

使用步驟
  1. 遇到難解的 bug，輸入 /diagnosing-bugs 並描述問題
  2. Reproduce：建立可穩定重現的測試案例
  3. Minimise：縮減重現步驟到最小集合
  4. Hypothesise：列出可能原因並排序
  5. Instrument：加入 logging/metrics 驗證假設
  6. Fix：實施修復
  7. Regression Test：加入回歸測試防止再發

code-review

/code-review
🤖 Model-invoked

雙軸 Code Review：Standards 軸（是否符合 repo 標準 + Fowler code smell）和 Spec 軸（是否忠實實作原始 issue/PRD）。以平行 sub-agents 執行，確保互不干擾。

使用步驟
  1. 完成一段程式後，可直接輸入 /code-review
  2. 或提供參考點：/code-review since:abc1234（指定 commit SHA）
  3. 兩個平行 agent 分別從 Standards 和 Spec 角度 review
  4. 查看雙軸 review 報告
  5. 依建議修改後重新 review
⚖️ 平行執行確保兩個視角互不污染，得到更客觀的 review 結果

prototype

/prototype
🤖 Model-invoked

建立用完即丟的 Prototype 來回答設計問題。邏輯/狀態問題 → 可執行的終端機 app；UI 問題 → 多種截然不同的 UI 變體可在同一路由切換比較。

使用步驟
  1. 輸入 /prototype 並說明想驗證的設計問題
  2. 指定 prototype 類型：邏輯驗證或 UI 比較
  3. Agent 快速建立 throwaway prototype（不求完美）
  4. 實際測試 prototype，收集設計決策依據
  5. 得到答案後，丟棄 prototype，用所學設計正式實作
🗑️ Prototype 的目的是「學習」而非「交付」，勿將 prototype 程式碼直接用於生產

research

/research
🤖 Model-invoked

對問題進行系統化調查，依靠高信任度的主要來源（primary sources），並將發現以有引用的 Markdown 檔案存入 repo。以背景 agent（background agent）執行。

使用步驟
  1. 輸入 /research 並描述調查問題
  2. 指定可信來源優先順序（官方文件、RFC、論文等）
  3. Agent 在背景執行調查，不阻塞主工作流程
  4. 調查完成後，在 repo 內生成有引用的 Markdown 報告
  5. 引用報告內容作為後續決策依據

domain-modeling

/domain-modeling
🤖 Model-invoked

主動建立和精化專案的領域模型。挑戰術語是否符合 glossary、以 edge-case 場景壓力測試，並即時更新 CONTEXT.md 和 ADRs。

使用步驟
  1. 通常由 /grill-with-docs 自動觸發，也可直接輸入
  2. 提供要建模的業務領域或概念
  3. Agent 挑戰現有術語，確保一致性
  4. Agent 用 edge cases 測試模型的完備性
  5. 自動更新 CONTEXT.md 的 glossary 章節

codebase-design

/codebase-design
🤖 Model-invoked

設計「深度模組」（Deep Modules）的共享紀律與詞彙：透過小型介面（interface）暴露大量行為，放置在乾淨的接縫（seam），並可透過該介面進行測試。

使用步驟
  1. 設計新模組或重構時觸發 /codebase-design
  2. Agent 詢問模組應暴露的最小介面
  3. Agent 協助識別模組的天然接縫（seams）
  4. 制定測試策略，確保可透過介面測試
  5. 輸出設計決策，納入 ADR
📚 基於 John Ousterhout《A Philosophy of Software Design》的深度模組原則

resolving-merge-conflicts

/resolving-merge-conflicts
🤖 Model-invoked

逐 hunk 處理進行中的 git merge 或 rebase 衝突，透過追溯每一方的原始意圖（primary source）來解決，最後完成 merge/rebase 操作——絕不 --abort。

使用步驟
  1. 遇到 merge conflict 後，輸入 /resolving-merge-conflicts
  2. Agent 逐一檢視每個衝突 hunk
  3. Agent 追溯兩側各自的原始意圖（相關 commit、issue）
  4. 依意圖優先順序解決衝突，非機械式合併
  5. 所有 hunk 解決後，完成 merge 或 rebase 操作
🚫 原則：永遠不使用 --abort，面對問題解決問題

Productivity Skills

通用工作流程工具，不限程式開發場景

grill-me

/grill-me
👤 User-invoked

非程式場景的深度訪談版本。對任何計劃或設計進行無情的連環提問，直到決策樹的每個分支都解決為止。/grill-with-docs 的輕量版，不帶文件生成。

使用步驟
  1. 輸入 /grill-me 並描述你的計劃、想法或問題
  2. Agent 展開連環提問，不放過任何模糊地帶
  3. 誠實回答每個問題（模糊答案會被追問）
  4. 訪談結束後，你對自己的計劃有清晰的全貌
💡 非常適合在寫文章、做決策、規劃旅程等非工程場景使用

handoff

/handoff
👤 User-invoked

將當前 conversation 壓縮成交接文件（Handoff Document），讓另一個 agent（或新 session）可以無縫接續工作。解決 context window 用盡的問題。

使用步驟
  1. 當 session 快達到 context limit 時，輸入 /handoff
  2. Agent 分析整個對話，提煉關鍵狀態和未完成工作
  3. Agent 生成結構化的交接文件（包含：當前狀態、決策紀錄、待辦事項、相關檔案）
  4. 開啟新 session，貼入交接文件作為開場
  5. 新 session 的 agent 可立即了解上下文並繼續工作

teach

/teach
👤 User-invoked

透過多個 session 教授你新技能或概念，以當前目錄作為有狀態的教學工作空間。Session 間保持學習進度。

使用步驟
  1. 輸入 /teach [主題]，例如：/teach TypeScript generics
  2. Agent 評估你的現有知識水準
  3. Agent 制定個人化教學計劃
  4. 每個 session 完成一個學習單元，並在目錄留下學習記錄
  5. 下次輸入 /teach，從上次中斷處繼續
📚 利用本地目錄作為持久化的學習狀態，跨 session 不遺失進度

writing-great-skills

/writing-great-skills
👤 User-invoked

撰寫和編輯 Skills 的參考指南。包含讓 skill 可預測的詞彙和原則。想要自己擴充或修改 skills 時的必讀資料。

使用步驟
  1. 想寫新 skill 時，先輸入 /writing-great-skills 閱讀指南
  2. 了解 User-invoked vs Model-invoked 的設計原則
  3. 學習 skill 的結構格式（SKILL.md 的撰寫慣例）
  4. 按照指南的詞彙和模式撰寫你的自訂 skill
  5. 請 agent 使用 /writing-great-skills 的標準 review 你寫的 skill
✍️ 使用 skills.sh 安裝的使用者可直接修改 SKILL.md 檔案客製化行為

grilling

/grilling
🤖 Model-invoked

無情訪談使用者，直到計劃、決策或想法的每個決策樹分支都解決為止。這是 /grill-me 和 /grill-with-docs 背後共用的可重用循環邏輯。

使用步驟
  1. 通常由 /grill-me 或 /grill-with-docs 自動觸發
  2. 也可作為 sub-skill 被其他自訂 skill 呼叫
  3. 接受要訪談的主題或文件作為輸入
  4. 執行結構化訪談直到問題完全解決
🔧 撰寫自訂 skill 時可引用此 skill 獲得訪談能力
